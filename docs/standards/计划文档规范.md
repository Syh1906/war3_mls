# 计划文档规范

## 适用范围

- `docs/plans/*.md` 中进行中或待定的正式计划正文。
- `docs/plans/archive/**/*.md` 中已归档的计划正文。

## 文件命名

- 进行中 / 待定计划：`NN-主题推进计划.md`，例如 `01-兑换码系统推进计划.md`。
- `NN` 为两位十进制编号，从 `01` 开始递增；已有最大编号为 `07` 时，下一个计划编号必须是 `08`。
- 编号一经分配不得复用，即使原计划取消、合并或归档也不回收。
- 主题名应短而稳定，默认不超过 18 个中文字符；不要使用空格、日期、临时状态词或英文 slug。
- 文件名必须保留“推进计划”后缀，避免把内部推进计划误写成普通说明文档。
- 归档计划：`docs/plans/archive/YYYYMMDD-NN-主题推进计划.md`，保留原编号并增加完成日期。

## 编号分配规则

1. 新建计划前先读取 `docs/plans/README.md` 和 `docs/plans/archive/README.md`，确认当前最大编号。
2. 如果归档目录中存在更大的历史编号，也要继续向后递增。
3. 如果多个计划同时创建，按实际落盘顺序连续分配编号。
4. 取消或废弃的计划仍保留编号，在计划正文和索引中标注状态，不删除编号历史。
5. 如果项目画像指定了项目级前缀，可使用 `<PREFIX>-NN-主题推进计划.md`，但归档时仍必须保留 `NN`。

## 历史无编号计划迁移

项目接入治理前已经存在的无编号计划，不要静默重命名或删除。先在 `docs/plans/README.md` 标记为“历史未编号计划”，再按用户意图选择：

1. 继续推进时，分配下一个 `NN`，重命名为 `NN-主题推进计划.md`，并同步所有引用。
2. 已经完成时，先验收并回填长期事实，再归档为 `YYYYMMDD-NN-主题推进计划.md`。
3. 暂不处理时，保留原文件名，但 README 必须说明它是历史遗留，不作为新计划命名范例。

## 固定章节顺序

1. `# 标题`
2. `## 元信息`
3. `## 概述`
4. `## 最终目标`
5. `## 范围边界`
6. `## 当前代码基线`
7. `## 当前缺口`
8. `## 全局冻结契约`
9. `## 首轮待冻结决策`
10. `## 里程碑总览`
11. `## 阶段详述`
12. `## 风险`
13. `## 相关文件`
14. `## 文件变更清单`
15. `## 变更记录`

## 结构要求

- `元信息` 使用 `项目 / 值` 表头，至少包含：`计划编号`、`文档类型`、`当前状态`、`当前阶段`、`首次建立`、`最后更新`、`计划范围`、`明确不含`、`关联目录`。
- `范围边界` 必须拆成 `### 本期范围` 和 `### 明确不在本期`，不要只用一张简单包含 / 不包含表。
- `当前代码基线` 使用 `领域 / 当前事实 / 证据` 表头；没有证据的结论不能写进基线。
- `全局冻结契约` 用于记录后续阶段不得偏离的稳定口径；没有冻结项时写“待冻结”，不要删除章节。
- `首轮待冻结决策` 用复选框维护已确认和待确认的关键决策。
- `里程碑总览` 使用 `里程碑 / 目标 / 截止日期 / 负责人 / 状态` 表头。
- `阶段详述` 下每个阶段至少包含 `目标`、`阶段冻结契约`、`待办清单`、`验收规格`、`阶段出口条件` 和 `依赖`。
- `文件变更清单` 拆成 `### 新建`、`### 修改`、`### 删除` 三组；即使暂无删除项，也要保留删除组并写“无”。
- `变更记录` 使用 `日期 / 变更内容 / 影响范围 / 原因 / 审批人` 表头。

## 进度同步规则

- 每次勾选待办、切换阶段、冻结结论、调整范围、消化 blocker 后，立即同步：

  - `当前状态`
  - `当前阶段`
  - `最后更新`
  - 里程碑状态
  - `变更记录`

- `当前状态` 只能使用项目画像登记的稳定枚举；默认枚举为 `待定`、`进行中`、`阻塞`、`已完成`、`已取消`。
- 代码或文档已经完成时，不能让计划继续停留在旧阶段或旧状态。

## 归档规则

1. 计划进入 `已完成` 前，先确认验收项已完成并记录验证证据。
2. 先把长期事实回填到正式 docs，例如项目总览、架构、API、用户指南、LLM 约束或目录 README。
3. 再把计划正文移动到 `docs/plans/archive/YYYYMMDD-NN-主题推进计划.md`。
4. 在归档正文的元信息中保留原 `计划编号`，并把 `当前状态` 更新为 `已完成`。
5. 从 `docs/plans/README.md` 的当前计划列表移除该计划。
6. 在 `docs/plans/archive/README.md` 中新增归档记录，至少包含编号、计划、完成日期和回填说明。
7. 归档后执行 docs impact review，确认总导航、相关正文和 `docs/llm` 是否需要同步。

## 禁止项

- 把长期事实只留在计划里，不回填正式 docs。
- 代码已经完成但计划状态未同步。
- 新建无编号计划，或归档时丢失原计划编号。
- 仅移动文件而不同步 `docs/plans/README.md` 和 `docs/plans/archive/README.md`。
