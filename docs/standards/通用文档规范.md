# 通用文档规范

## 元信息

| 项目 | 值 |
| --- | --- |
| 适用范围 | `docs/` 下全部正式 Markdown 文档 |
| 不适用 | 临时记录、生成产物、一次性排查草稿 |
| 优先级 | 项目内 docs 规范与模板 > `docs-governance` 内建规范与模板 > `doc-writing` |

## 读取顺序

1. 先判断文档类型，不要直接开写。
2. 先读 `docs/standards/README.md`，再读对应类型规范。
3. 再读 `docs/templates/` 下对应模板，按当前项目骨架落稿。
4. 同步检查当前正文、目标目录 `README.md`、`docs/README.md`、计划状态与归档索引、API 文档与接口变更记录是否也要更新。
5. 同步检查 `docs/standards/*.md`、`docs/templates/*.md`、用户指南和 `docs/llm` 是否也要更新。

## Docs Impact Review 门禁

1. 只要改了代码、配置、脚本、接口、UI、行为、目录结构、测试口径或计划状态，就必须评估是否要同步正式 docs。
2. docs impact review 不是只在“正在写文档”时才做，而是任务收尾前的全局门禁。

## 共通硬规则

1. 正式 docs 只写长期事实、稳定边界、执行口径和长期有效的阅读入口。
2. 先证据后结论，不能把猜测写成事实。
3. 第一屏必须回答：给谁看、解释什么、不解释什么。
4. 结构化信息优先用表格、清单、图示。
5. 历史样式只有在项目治理层显式登记后才能沿用；未登记旧样式不能覆盖正式规范。

## 完成检查

- 文档类型判断正确，已读取对应类型规范。
- docs impact review 已执行。
- 当前正文、README、`docs/standards/*.md`、`docs/templates/*.md`、计划状态与归档索引、API 文档与接口变更记录、用户指南和 `docs/llm` 已做同步检查。
