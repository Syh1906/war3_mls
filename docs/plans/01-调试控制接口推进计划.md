## 概述

当前模拟器已经具备 HTTP 服务、房间管理和 Lua 运行时基础能力，但本地调试还缺少面向房间日志、房间重启和服务重启方案确认的稳定控制接口。这份计划先把调试控制能力收敛为“日志拉取、房间重启、服务重启方案确认、接口文档同步”的首版闭环，再进入实现。

## 最终目标

交付本地调试控制接口首版：开发者和 LLM 可以通过 HTTP 拉取房间日志、请求房间重启，并明确服务重启是否由模拟器进程内接口承担。接口实现后同步 HTTP 接口参考和验证口径，确保后续调试任务不依赖口头约定。

## 范围边界

### 本期范围

- 日志拉取接口：返回指定房间日志，并支持基础查询参数。
- 房间重启接口：重建指定房间，并返回新房间标识或稳定结果语义。
- 服务重启方案确认：明确是否需要独立服务级接口，以及是否由外部工具重新拉起进程。
- HTTP 接口文档同步和 `cargo check` 验证口径。

### 明确不在本期

- 线上运营鉴权、多用户权限和审计日志。
- 访问令牌轮换、长期令牌管理和公网安全策略。
- 完整 Web 控制台 UI。
- 跨进程守护、自动拉起和部署级进程管理。

4. HTTP 接口参考还没有记录这些调试控制能力。
5. 尚未执行 `mls-sim-rs/` 下的 `cargo check` 验证。

## 全局冻结契约

> 以下条款一经确认即为冻结项，后续阶段不得偏离，除非走变更记录流程。

1. 本期只做本地调试控制能力，不提前引入线上运营鉴权和公网安全策略。
2. 房间级重启和服务级重启必须拆开描述，不能把两类能力混成一个接口。
3. 接口实现后必须同步 `docs/api/HTTP接口参考.md` 和计划状态。
4. 涉及 Rust 代码后，验证命令在 `mls-sim-rs/` 下执行 `cargo check`。

## 阶段详述

### 阶段 1：接口契约确认

**目标**：冻结日志拉取、房间重启和服务重启方案的接口边界。

**阶段冻结契约**：
> 1. 先冻结路径、参数、返回结构和失败语义，再进入实现。
> 2. 服务重启如果不由模拟器内接口承担，必须明确外部工具的职责边界。

**待办清单**：
- [ ] 复核现有 HTTP 路由和房间模块，确认接口落点。
- [ ] 确认日志拉取的日志来源、分页或行数限制。
- [ ] 确认房间重启后返回原房间 ID、新房间 ID 还是状态对象。
- [ ] 确认服务重启方案是否纳入 HTTP API。

**验收规格**：
- [ ] 三类能力都有明确路径、参数、返回结构和失败语义。
- [ ] 不再依赖口头说明补齐接口边界。

**阶段出口条件**：接口契约已冻结，可进入实现。

**依赖**：当前 HTTP 服务和房间运行时结构。

### 阶段 2：接口实现

**目标**：实现日志拉取和房间重启接口，并按阶段 1 结论处理服务重启方案。

**阶段冻结契约**：
> 1. 实现必须沿现有 HTTP 路由和房间运行时结构扩展。
> 2. 不在本阶段引入线上运营鉴权和复杂权限系统。

**待办清单**：
- [ ] 新增日志拉取 handler。
- [ ] 新增房间重启 handler。
- [ ] 按冻结结论处理服务重启方案。
- [ ] 补齐必要错误响应。

**验收规格**：
- [ ] 日志拉取接口返回稳定结构。
- [ ] 房间重启接口可以重建房间并返回稳定结果。
- [ ] 不支持的服务重启方案有明确说明或错误响应。

**阶段出口条件**：本地调试接口可被调用。

**依赖**：阶段 1。

**目标**：同步 HTTP 接口文档和验证证据，完成计划收口。
