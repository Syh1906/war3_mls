## 概述

当前模拟器已经切换到原生 GUI 管理房间、控制台日志、出站事件、状态和设置，并保留 Bridge HTTP 服务给客户端登录、发送事件和轮询事件。本计划已完成首版本地自动化调试接口：GUI 继续承担人工调试，`/api/debug/*` 承担本地脚本、LLM 和外部工具的自动化调试入口。

## 最终目标

已交付本地自动化调试 HTTP 接口首版：开发者、LLM 和测试脚本可以通过 HTTP 读取房间日志、清空房间侧日志缓存、请求房间级重启；服务级重启明确由外部进程管理工具负责。HTTP 接口参考和用户指南已同步，验证口径已记录。

## 范围边界

### 本期范围

- 自动化日志拉取接口：从房间侧日志缓存读取指定房间日志，并支持基础查询参数。
- 自动化日志清空接口：清空指定房间的房间侧日志缓存，避免长时间调试后日志过长影响传输。
- 房间级重启接口：基于现有房间配置重建指定房间，并返回稳定结果语义。
- 服务级重启方案确认：明确是否由模拟器进程内 HTTP 接口承担，或交给外部工具重新拉起进程。
- HTTP 接口文档、用户指南和 `cargo check` 验证口径同步。

### 明确不在本期

- 线上运营鉴权、多用户权限和审计日志。
- 访问令牌轮换、长期令牌管理和公网安全策略。
- Web 控制台 UI。
- 跨进程守护、自动拉起和部署级进程管理。
- 替代现有原生 GUI 的人工调试能力。

## 当前缺口

1. 已冻结调试控制路由命名空间为 `/api/debug/*`，与 `/api/bridge/*` 客户端契约分离。
2. 已冻结日志拉取接口只读取房间侧 `log_buffer`，支持 `limit`、`level`、`q`、`since` 查询参数。
3. 已冻结日志清空接口只清理房间侧 `log_buffer`，不影响 GUI 本地显示列表。
4. 已冻结房间重启语义为创建新房间 ID，并返回 `old_room_id`、新 `room_id` 和 `status`。
5. 已冻结服务级重启不进入模拟器进程内 HTTP API，由外部工具负责重新拉起进程。
6. HTTP 接口参考和用户指南已记录这些自动化调试能力。

## 全局冻结契约

> 以下条款一经确认即为冻结项，后续阶段不得偏离，除非走变更记录流程。

1. 本期只做本地自动化调试能力，不提前引入线上运营鉴权和公网安全策略。
2. Bridge 客户端通信契约保持稳定；新增调试控制路由不得改变现有 `/api/bridge/*` 请求和响应语义。
3. GUI 人工调试入口与 HTTP 自动化调试入口并行存在；HTTP 接口不替代原生 GUI。
4. 房间级重启和服务级重启必须拆开描述，不能把两类能力混成一个接口。
5. 接口实现后必须同步 `docs/api/HTTP接口参考.md`、相关用户指南和计划状态。
6. 涉及 Rust 代码后，验证命令在 `mls-sim-rs/` 下执行 `cargo check`。
7. 调试控制路由命名空间固定为 `/api/debug/*`，首版实现继续放在 `mls-sim-rs/src/bridge.rs` 中。
8. 日志拉取路径固定为 `GET /api/debug/rooms/{room_id}/logs`，默认返回最近 100 条匹配记录，最多 500 条。
9. 日志清空路径固定为 `POST /api/debug/rooms/{room_id}/logs/clear`，只清空房间侧日志缓存。
10. 房间重启路径固定为 `POST /api/debug/rooms/{room_id}/restart`，成功后返回新房间 ID。
11. 服务级重启路径固定为 `POST /api/debug/service/restart`，返回 `501` 并提示外部工具负责进程重启。
