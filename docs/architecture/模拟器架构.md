# 模拟器架构


## 概述

模拟器是一个 Rust 单二进制程序，内部包含：

- **原生 GUI**（egui/eframe）：管理房间、日志、出站事件、状态和设置
- **Bridge HTTP 服务器**（axum）：提供客户端登录、发送事件、轮询事件和健康检查
- **Lua 运行时**（mlua，Lua 5.3）：每个房间一个独立的 Lua VM
- **存档系统**：JSON 文件持久化，自动读写

## 线程模型

```
┌────────────────────────────────────┐
│  原生 GUI 主线程                    │
│  展示房间、日志、事件、状态和设置    │
└──────────┬─────────────────────────┘
           │ 读写共享房间管理器
┌──────────▼─────────────────────────┐
│  tokio 异步运行时                   │
│  处理 Bridge HTTP 请求              │
└──────────┬─────────────────────────┘
           │ 通过 channel 发送命令
     ┌──────▼──────┐
     │  房间线程    │  每个房间一个独立 OS 线程
     │  Lua VM     │  独占 lua_State，线程安全
     │  事件循环    │  50ms 间隔轮询命令队列
     └──┬──────────┘
        │ 广播和队列
        ├─→ GUI 控制台和出站事件页
        └─→ Bridge 轮询队列
```

- GUI 操作和 Bridge HTTP 请求通过 channel 向房间线程发送命令（发事件、停止等）
- 房间线程独占 Lua VM，所有 Lua 调用都在同一线程执行
- 日志和出站事件通过广播供 GUI 展示；客户端通过 Bridge 轮询读取自己的出站事件队列

## Lua API 注入

每个房间的 Lua VM 在启动时注入以下全局 API：

| 分类 | API |
| --- | --- |
| 日志 | `Log.Debug/Info/Error` |
| 定时器 | `Timer.After`、`Timer.NewTicker` |
| 事件 | `RegisterEvent`、`UnregisterEvent` |
| 玩家查询 | `MsGetPlayerName/MapLevel/MapExp/PlayedTime/TestPlayTime/PlayedCount` |
| 房间查询 | `MsGetRoomStartTs/LoadedTs/GameTime/PlayerCount/ModeId` |
| 道具 | `MsGetPlayerItem`、`MsConsumeItem` |
| 存档 | `MsGet/SaveScriptArchive`、`MsGetCommon/Read/CfgArchive`、`MsSetReadArchive` |
| 控制 | `MsSendMlEvent`、`MsEnd` |
| JSON | `json.encode`、`json.decode`（纯 Lua 实现，自动注入） |
| 模块 | `require`（自定义实现，支持相对路径和 `.` 分隔） |

详细参数和返回值见 [Lua 接口参考](../api/Lua接口参考.md)。

## 自定义 require

模拟器重写了 Lua 的 `require` 函数：

- `require("event.ms_event_api")` → 加载 `event/ms_event_api.lua`
- `require("../map/xxx")` → 支持相对路径
- 先从当前模块所在目录搜索，再从脚本根目录搜索
- 支持 `init.lua` 目录模块
- 模块缓存，防止重复加载

## 内置系统事件

房间生命周期会自动触发以下事件（需要通过 `RegisterEvent` 注册才能收到）：

| 事件名 | 触发时机 | 数据 |
| --- | --- | --- |
| `_roomloaded` | 房间加载完成 | `{"players": [0, 1]}` |
| `_roomover` | 房间结束 | `{"reason": "GameEnd"}` |
| `_playerexit` | 玩家退出 | `{"reason": "Logout"}` |
| `_playerleave` | 玩家断线 | `{"reason": "Disconnect"}` |
| `_playerjoin` | 玩家重连 | `{"reason": "Connect"}` |

## 存档系统

**保存流程：**
1. 运行中 → Lua 脚本调用 `MsSaveScriptArchive` 等 API，立即更新内存并尝试落盘
2. 房间停止、GUI 关闭或进程正常退出 → 自动保存到 `archives/<脚本目录名>.json`

**读取流程：**
1. 创建房间 → 自动检查 `archives/` 目录是否有同名脚本的存档
2. 如果有，自动加载到玩家数据中
3. 配置文件中手动指定的存档优先级更高，不会被覆盖

## Bridge HTTP

Bridge HTTP 服务默认监听 `http://127.0.0.1:5000`，用于客户端或测试脚本接入：

```text
POST /api/bridge/login
POST /api/bridge/event
GET  /api/bridge/poll/{room_id}/{player_index}
GET  /api/bridge/rooms
POST /api/bridge/config
GET  /api/health
```

GUI 内部不依赖浏览器或 WebSocket；日志和出站事件直接从房间广播通道读取。

## 日志限制

与平台一致的日志频率限制：

- 100 秒内最多 1000 条日志
- 超限后熔断，下一个周期恢复
- 单条日志最大 2000 字节
